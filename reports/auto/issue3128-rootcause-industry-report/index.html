Issue 3128「parseProgram 段错误」根因分析与业界对照调研

TL;DR：这不是解析器正确性缺陷，而是 FFI 边界的栈预算错配——按 8MB 主线程栈编写的 C++ 递归下降解析器，被 std.ast 经 C2N stub 放到 128KB 仓颉协程栈上执行。 修复收敛为 stdlib 单仓 7 文件（runtime!1331），核心机制为「按输入定栈的专用解析线程 + 热点解码迭代化 + 残余转类型化异常」， 对 cjc 行为零改动；编译器侧 256 层深度上限方案（compiler!1625，已关闭）经门禁实证既不充分也不必要。 该组合的每一件在 clang / rustc / Roslyn / CPython / Go / PostgreSQL 中均有直接对应物。

一、根因分析

1.1 现象

用户程序对 11 个 .cj 文件循环调用 std.ast.parseProgram，在深嵌套输入上进程直接 SIGSEGV (core dumped)。issue 附带的崩溃堆栈明确指向库内 native 帧： 

#0  Cangjie::ParserImpl::ParseQualifiedType()   ← libcangjie-std-ast.so（C++ 解析器）
#1  CJ_MCC_C2NStub                               ← libcangjie-runtime.so（FFI 入口桩）
#2  parseProgram → parse(ffi.cj:77)              ← 仓颉侧 std.ast
#5  user main

1.2 因果链（五层）

  1. std.ast 复用了编译器的 C++ 递归下降解析器。 编译器 Parse 模块被直接编进 libcangjie-std-ast.so，仓颉侧经 ast_api.cpp 的 CJ_AST_Parse* 系列 extern 入口调用。这份解析器代码自诞生起的运行环境假设是 cjc 的 8MB 主线程栈，递归深度从未被显式约束——在 cjc 里这从来不是问题。
  2. C2N FFI 让 native 代码直接跑在协程栈上。 仓颉 cjnative 的 C2N stub（runtime/src/arch/<arch>/C2NStub.S，已核对 x86_64_linux / aarch64_macos 两版汇编）不切换栈：native 函数就在发起调用的 cjthread 协程栈上执行， Linux 默认 128KB（CjScheduler.cpp:224），macOS 默认 1MB。 作为对照，Go 的 cgo 在设计上禁止这样做——每次 C 调用都切到 OS 分配的 g0 系统栈，因为 C 代码没有栈增长 prologue、无法参与协程栈的协作式扩容。仓颉 C2N 当前缺这一层，是根因中的架构性成分。
  3. 栈预算错配：64 倍的差距。 8MB 假设 ÷ 128KB 现实 = 64 倍。按实测帧开销折算成可承受的嵌套深度：
    递归路径触发形态（issue 复现文件）实测帧开销128KB 栈崩溃点8MB 栈（cjc）
    ParseType → ParseQualifiedType → ParseRefType425 层 S<S<…>>（regression_fuzz_0017.cj）≈0.5KB/层≈250 层（issue 主崩溃点）≈1.6 万层
    ParseExpr → ParseAtom → ParseReturnExpr2 万层 return 链（multi_return.cj）2–4KB/层≈40–60 层≈3000 层（cjc 在 2 万层上同样 ICE，rc139，有既有 LLT 记录）
    ParseUnaryExpr → ParseBaseExpr深 ! 前缀链（timeout_06/07.cj 族）≈0.5KB/层≈250 层≈1.6 万层

    帧开销测定方法：return 链 2000 层在 8MB 栈通过（≤4KB/层）、2 万层在 40MB 栈溢出（≥2KB/层），夹逼得 2–4KB/层；type 链由 issue 崩溃深度 ≈250 层 ÷ 128KB 反推 ≈0.5KB/层。

  4. C2N 入口栈检查对递归 native 函数无效 → 不可捕获的 SIGSEGV。 CJ_MCC_C2NStub 入口确实有一次栈检查（汇编核对：sp ≤ CJThreadStackGuardGet() + cpStackSize 时 调 MCC_ThrowStackOverflowError，抛可捕获的 StackOverflowError）。但 cpStackSize 是 编译期对单次调用的静态栈需求估计，而 ParseType 的递归深度由运行时输入决定：入口检查放行后， native ParseType→ParseQualifiedType→ParseType→… 是纯 C++ 递归，不再经过任何 stub、没有后续栈检查，也无法触发协程栈增长 （cjthread 有 stackGrow 机制，但只有托管函数的栈增长 prologue 能触发它，见上一层 Go cgo 类比）。 于是逐层吃栈，越过 mprotect 保护的 stackGuard 保留区（cjthread.cpp:193/1333）→ SIGSEGV，不可捕获。 更糟的是无 sigaltstack 时信号处理器在已溢出栈上二次崩溃，表现为无任何输出的静默退出（rc 132）——连 issue 里那种崩溃堆栈都打不出来。
  5. 次生问题：解析成功后，反序列化在仓颉侧重演同样的递归。 native 解析产出 flatbuffer，仓颉侧 createBinaryExpr / createReturnExpr / createUnaryExpr / createRefType 逐层递归解码——同样跑在 128KB 协程栈上。左结合长二元链（2 万个 +，parameters_arguments_in_one_func.cj）还会让 native 侧的 Walker::Walk / SerializeBinaryExpr 先崩（Bus Error）。这层是可捕获的 StackOverflowError， 严重度低一级，但同样让合法输入无法解析。

1.3 一句话根因与定性

根因：「按 8MB 栈编写的递归代码」与「128KB 的实际执行栈」之间的预算错配，错配点在 std.ast 的 FFI 边界。 定性：这是 std.ast 的部署环境缺陷，不是解析器的正确性缺陷——同一份解析器在 cjc 里对 issue 全部输入工作正常。 因此修复归属在 std.ast，而非给解析器加深度上限（上限治不了 40 层就崩的 return 链，还会在补丁版本里收窄 cjc 接受的语言）。

1.4 为什么 256 层深度守卫方案（compiler!1625）被否决

检验项结论证据
充分性不充分守卫按 cjc 的 8MB 栈校准；128KB 协程栈在 ≈250 层（type 链）/ ≈40 层（return 链）就崩，守卫没有机会触发——Linux 门禁 18 个用例 SIGSEGV 实证
必要性不必要大栈线程合入后崩溃消失，无需任何深度上限
兼容性回退1.1.0 能编译的 >256 层嵌套代码在 1.1.x 补丁版本突然报错；业界硬上限（GCC -ftemplate-depth、clang -fbracket-depth、SQLite）均为文档化 + 发布版引入 + 按栈校准，无补丁版本悄悄加限的先例

二、修复策略与验证（runtime!1331，7 文件，全部在 std.ast 内）

机制对应根因层实现要点
按输入定栈的专用解析线程
ast_api.cpp		 第 2、3 层（栈预算） 8 个 extern 入口整体（含 ParseWithError 重解析与 NodeWriter 序列化）调度到专用线程；栈 = clamp(tokenCount × 8KB, 8MB, 2GB)——token 数从序列化流头部 uint32 零成本读出，是嵌套深度的数学上界（每层至少消耗 1 token），8KB/token 对实测最贵路径（return 链 2–4KB）留 2 倍余量；虚拟内存惰性提交，扁平大输入只多预留不拒绝；≤32 token 走 inline 快路径（宏展开热路径零线程开销）；创建失败回退 8MB，再失败返回错误 ParseRes（仓颉侧抛 ParseASTException），不再冒险 inline
热点解码迭代化
parse_expr.cj / parse_type.cj		 第 5 层（解码递归） createBinaryExpr（左结合链）、createReturnExpr / createUnaryExpr（右嵌套链）、createRefType（单参数泛型链）改为「迭代收集 → 自底向上重建」；逐层位置信息与原递归路径精确等价（独立评审逐项核对确认）
残余转类型化异常
ffi.cj decodeAST + 9 个公共 API		 第 5 层兜底 未迭代化形态（多参数泛型嵌套、深括号链）的 StackOverflowError 统一转为 ParseASTException("…nests too deeply to decode; increasing the cjthread stack size (cjStackSize) may allow deeper inputs.")——守卫在前、提示在后（PostgreSQL HINT 模式）

三、业界对照调研：五个处理家族

家族一：检测栈余量，不足时换大栈继续 —— clang / rustc / Roslyn

clang（Basic/Stack.h）：DesiredStackSize = 8MB， isStackNearlyExhausted() 检测 + runWithSufficientStackSpace() 先发 warn_stack_exhausted 诊断、再换大栈执行。设计动机（D66361）：递归操作「实践上无法保证不爆栈，崩溃时用户体验极差」。 关键事实：2019–2025 年其实现就是经 CrashRecoveryContext 起一个新线程换栈——与本修复同构—— PR #133173（2025）才优化为直接分配栈段。机制还被扩展到反序列化层 （PR #79875），对应本修复把序列化一并罩进大栈线程。

rustc：ensure_sufficient_stack 包装 stacker，递归热点插桩，余量 <100KB 红区时分配 1MB 新栈段（PR #55617）。 反面教材 #128422（调试全记录）： 拍脑袋的 1MB 栈段对真实生成文件不够，照样段错误——印证「按实测帧开销 + 余量定值」而非猜测的必要性； stacker 社区某失败案例的 workaround 恰是「起一个 8MB 栈的新线程」。

Roslyn：.NET 的 StackOverflowException 不可捕获（与 native SIGSEGV 同性质），故必须前置检测—— VisitExpressionWithStackGuard + EnsureSufficientExecutionStack() 抛可捕获异常，最终落为 ERR_InsufficientStack 编译诊断（#70381）。

家族二：解析时栈限位 → 类型化错误 —— V8 / SpiderMonkey / CPython

JS 引擎在解析器中做栈指针限位检查，深嵌套源码抛 JS 层 RangeError（SpiderMonkey：InternalError: too much recursion），进程不死。 CPython 的演进最完整：旧 pgen 解析器固定 MAXSTACK 报 "parser stack overflow"； 3.12 起（gh-91079 / PR #96510）把 C 栈防护与 Python 递归计数解耦； 3.14 改为真实栈指针对比软/硬限位（pycore_ceval.h），超限抛可捕获 RecursionError。 对应本修复的 decodeAST 兜底，也是 dev 议题「栈余量探测」的原型。

家族三：文档化的硬深度上限 —— GCC / clang / SQLite / Jackson / simdjson / MSVC

clang -fbracket-depth=256、GCC -ftemplate-depth=900 （默认 900 而非标准建议的 1024，因为「编译器可能在 1024 前先耗尽栈」——上限必须按实际栈校准，这正是 1625 的 256 在 128KB 栈上失效的教训）、 SQLite SQLITE_MAX_EXPR_DEPTH=1000（运行时只可调低）、 MSVC C1026。 最具参考价值的是 Jackson：2.15 在小版本中加入 maxNestingDepth=1000 作为不可信输入的安全加固（后关联 CVE-2025-52999），条件是宽松到零误伤的默认值 + 可配置 + 超限抛类型化异常——这是 dev 分支深度守卫提案可借鉴的引入条件。

家族四：运行时层 FFI 栈切换 —— Go cgo（架构终态）

Go 运行时从设计上禁止 C 代码跑在 goroutine 栈上： goroutine 栈靠编译器 prologue 协作增长，C 代码无 prologue 不能参与，故 每次 cgo 调用切到 OS 分配的 g0 系统栈执行、返回再切回 （golang-nuts：在 goroutine 栈上跑 C 是 footgun）。 仓颉 C2N 让 native 解析器直接跑在 128KB 协程栈上，正是 Go 刻意规避的反模式——这是「runtime 在 C2N 层提供大栈切换原语」作为终态方案的最有力背书；届时 std.ast 的局部方案可自然退役。

家族五：「提示用户调栈参数」作为正式机制 —— PostgreSQL / JVM 系（有前提条件）

PostgreSQL：check_stack_depth() 守卫先拦下溢出保住进程，再报 ERROR: stack depth limit exceeded + HINT: Increase "max_stack_depth"… （实例）； 且 Heroku 明确拒绝放开该参数（须与 OS ulimit 联动，调错会崩后端）——全局栈旋钮的运维风险有实证。 JVM 系（scalac 官方建议给 64MB 栈、-Xss）把调栈作为正式答案， 前提是 StackOverflowError 可捕获、崩溃有完整栈迹。 关键结论 业界从无「裸崩溃 + 让用户猜环境变量」的先例：提示永远建立在守卫或可捕获性之上。 故 cjStackSize 不能作为本 issue 的修复（主崩溃是 native SIGSEGV，无报错路径；所需值随输入线性增长，multi_return.cj 需 ≥64MB 全局生效）； 但在 decodeAST 的类型化异常消息中附 cjStackSize 提示（解码侧调大确实有效）符合 PostgreSQL HINT 模式，已落地。
