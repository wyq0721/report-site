01 项目背景

ICE 问题阻塞客户编译链路，且定位高度依赖现场材料

当前客户反馈通常只有编译器版本、架构平台和 ICE 日志，能初步判断阶段，但不足以完整复现和定位。

客户侧影响

ICE 会直接中断编译流程，客户无法通过普通诊断信息自行定位，容易阻塞交付和验证。

研发侧痛点

缺少最小复现用例时，开发难以从 release 日志直接还原触发路径，需要多轮补充材料。

协同侧成本

支撑、客户、开发之间反复沟通，等复现材料齐全后才能切 debug 版本或符号化环境分析。

立项判断：本课题不承诺短期消灭所有 ICE，而是先解决“能否更快启动有效分析”的过程质量问题。
02 现状访谈

访谈结论集中在信息缺口和流程口径不一致

访谈结论用于识别支撑、开发、测试质量和工具链环节的共同痛点。

角色主要痛点当前缺失信息期望改进
客户支撑 客户通常能提供截图、版本和错误日志，但完整工程、依赖和复现步骤获取困难。 统一问询模板、必填字段、脱敏材料边界。 有一份支撑侧可直接使用的 ICE 问询清单。
编译器开发 只有 ICE 错误码和阶段信息时，难以判断根因；缺少最小复现用例时只能做粗分流。 完整命令、最小复现、调用栈、编译选项、依赖和缓存状态。 首轮材料能支持判断 parser / sema / CHIR / codegen / driver 归属。
测试 / 质量 历史 ICE 修复后用例沉淀不稳定，过程指标缺少固定统计口径。 问题闭环状态、复现材料质量、回归用例关联关系。 建立 ICE 样本台账和案例库字段规范。
工具链 / 发布 release 包日志和栈信息有限，debug 包、符号化和 crash report 流程不够标准化。 符号文件匹配、crash dump、bug report 打包字段。 明确 release/debug 版本符号化和调用栈解析流程。
03 当前流程与卡点

从客户反馈到开发分析存在多轮等待

当前流程中最容易卡住的是“信息是否足够”和“是否能复现”。

1客户反馈版本、架构平台、ICE 日志
2支撑收集补充环境、场景和工程信息
3开发初判判断阶段和责任模块
4补充材料索要最小复现、命令、依赖
5本地复现对齐 SDK、平台、构建参数
6debug 分析切换符号或 debug 版本定位

卡点 1

首轮信息只能说明“哪里挂了”，不能说明“为什么挂”。

卡点 2

没有最小复现用例时，开发分析容易停在现象确认阶段。

卡点 3

历史相似问题无法快速检索，经验复用依赖个人记忆。

04 选题理由与边界

选题聚焦“定位启动效率”，避免范围过大

ICE 根因分散在多个编译阶段，不适合作为一个 QCC 周期内全部解决的目标。

本课题解决

  • ICE 首轮信息不完整
  • 问题分流和归属判断慢
  • 缺少统一采集模板和问询清单
  • 无源码场景下推进路径不清晰
  • 历史 ICE 案例复用不足

本课题不解决

  • 一个周期内消除所有 ICE
  • 所有编译器模块根因修复
  • 客户工程完整复现能力建设
  • 大规模编译器架构重构
  • 替代模块自身缺陷修复流程
推荐主题：提升仓颉编译器 ICE 问题定位启动效率。

08 信息完整率口径

当前首轮信息只能支持初判，不足以定义为完整

用户输入的现状：当前通常只有编译器版本、架构平台、ICE 日志和挂掉阶段；理论上需要最小复现用例才算完整。

字段当前可获得性是否计入完整标准说明
编译器 / SDK 版本通常有必填用于定位版本窗口和候选修复。
架构平台 / 目标平台通常有必填用于判断平台相关问题。
ICE 日志和挂掉阶段通常有必填用于初判 parser / sema / CHIR / codegen / driver。
完整编译命令 / 构建入口经常缺失必填影响复现和选项判断。
最小复现用例经常缺失完整判定核心项没有该项只能视为“可初判”，不能视为完整。
依赖、缓存、工程结构说明经常缺失选填增强用于复杂工程或增量相关问题。
建议口径：首轮信息完整 = 版本 + 平台 + ICE 日志/阶段 + 完整命令 + 最小复现用例。缺少最小复现时，最多计为“可初判”。
09 目标设定

目标聚焦首轮信息质量和定位启动效率

目标聚焦 QCC 小组可控的过程指标，避免把跨模块根因修复混入本课题范围。

指标当前基线目标值说明
首轮信息完整率待统计提升到 60%完整标准包含最小复现用例，目标值以可达成和可执行为原则。
可初判编译阶段比例已有版本/平台/阶段日志达到 85%当前已有阶段日志，目标应高于完整率。

10 典型案例

案例显示最小复现和完整命令能显著降低定位门槛

以下案例基于公开 issue 信息和立项场景整理，可在正式评审前替换为内部真实案例。

案例 A：信息相对完整

问题：#917 release/1.0 部分用例 ICE 11。

  • 提供了 cjc 版本、目标平台、完整命令和具体测试用例路径。
  • 开发可直接进入复现和 debug 分析。
  • 说明最小复现/明确命令对定位启动帮助明显。

案例 B：客户现场类问题

问题：客户工程触发 ICE，只能提供版本、平台和 ICE 日志。

  • 缺少完整命令、依赖、缓存状态和最小复现。
  • 开发只能判断大致阶段，无法确认触发路径。
  • 需要支撑多轮补充材料，定位启动时间被拉长。
案例结论：模板需要区分“最低可分析信息”和“完整信息”，避免所有问题都等待完整工程后才启动分析。
11 可行性与预期成果

优先交付模板、SOP 和样本台账

这些成果不依赖大规模编译器改造，QCC 小组可控。

数据可获取

GitCode issue、缺陷单、支撑记录、开发分析记录可形成样本台账。

改进可落地

采集模板、问询清单、triage SOP 可先于工具改造落地。

协同可组织

支撑、开发、测试、质量、工具链都有明确分工。

预期成果用途
ICE 标准信息采集模板统一客户反馈和支撑收集口径。
支撑侧问询清单减少二次补充信息比例。
开发侧 triage SOP提升可初判编译阶段比例。
ICE 样本台账和案例库字段沉淀历史问题、复用相似案例。
最小复现/脱敏材料指南降低客户提供材料成本。
12 评审关注问题

评审结论：主题成立，目标可度量

课题边界清晰，指标围绕信息完整、阶段初判、返工比例和启动时间展开。

可能追问建议回答
为什么不是“减少 ICE 数量”？ICE 根因分散，短周期内不可控；定位启动效率是跨模块可控过程指标。
没有客户源码，采集信息是否有用？有用。可先完成阶段分流、版本窗口判断、相似案例匹配和符号化分析准备。
目标值是否可靠？目标值以近 6 个月样本复核结果为基线，并按首轮材料质量进行校准。
工具改造超出周期怎么办？本周期先交付模板、SOP 和台账；crash report 工具增强作为标准化延伸项。
评审建议：同意立项，要求下一阶段完成样本人工复核、基线统计和目标值校准。
