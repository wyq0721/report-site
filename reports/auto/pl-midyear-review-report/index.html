2. 骨干锻炼、提升与团队优秀基因

已能讲的部分
  • 证据先行：PR 评审不凭印象，先拉 PR 元数据、diff、评论、门禁、issue，再给结论。
  • 复盘透明：!1815 合入后冒烟发现问题，继续补 !1832，并形成 AAR，而不是只做补丁。
  • 知识共享：把 Cangjie 经验沉淀成共享 memory、wikishare 文档、报告站点和专项 skill。
  • AI 必须验证：AI 代码生成不是“生成即可信”，而是接入 cjc、测试、codecheck、报告与复盘。
需要补充的部分
  • 骨干 A/B/C 各自承担了什么子领域，例如前端、运行时、测试、工具链、内存分析或报告平台。
  • 他们在 PR 评审、问题定位、技能沉淀、分享培训上的可举证案例。
  • 结果指标：独立解决问题数量、评审有效意见数量、主导交付件数量、新人带教效果。
  • 团队氛围反馈：是否形成主动复盘、主动沉淀、主动用 AI 验证的习惯。

3. 不足点与我没做到位的地方

度量不足

交付件很多，但业务收益、团队复用率、工时节省、缺陷拦截率还没有稳定度量。述职时容易显得“做了很多”，但影响力数字不够硬。

规模化不足

很多机制仍由我牵引或通过个人工作流推动，离“团队每个人自然使用同一套方法”还有距离。下半年要把机制交给骨干和流程。

质量闭环仍有漏网

!1815 的纰漏说明评审和自验证仍可能漏掉语义边界。需要把“边界用例设计、冒烟策略、AAR 反哺规则”进一步固化。

三、当前面临的关键挑战

AI 时代 + 仓颉编译器

挑战 1：通用模型不天然懂仓颉

仓颉是新语言，模型容易带入 C/JS/ArkTS/C++ 语法惯性，常见错误集中在 Option/match、包导入、API 参数与类型。编译器报错较准，但 agent 反复修正仍消耗时间。

我要做的改变：从“让 AI 写代码”升级到“让 AI 带着 Cangjie 上下文、真实 API 索引、cjc/LSP 快反馈、测试清单写代码”。

业务发展阻塞

挑战 2：复杂问题定位依赖现场材料

ICE、性能、内存和兼容性问题往往缺少现场代码、debug 版本、复现环境和足够日志。问题定位启动慢，容易多轮返工。

我要做的改变：围绕信息采集完整率、定位启动时间建立 QCC，沉淀 issue 模板、复现包、debug 指引和历史案例库。

团队能力阻塞

挑战 3：AI 工程能力没有完全组织化

已有共享记忆、agent、skill、报告站点和 hook，但使用方式、数据看板、责任人、质量门禁还不够团队化。

我要做的改变：把工具链变成团队制度：骨干 owner、每周 demo、每月复盘、skill 调用数据、报告复用数据、失败案例反哺。

质量与节奏阻塞

挑战 4：快交付和高质量之间张力变大

AI 提升了生成速度，也放大了误用 API、漏边界、上下文错配的风险。仓颉编译器又涉及前端、运行时、标准库、测试、文档多仓协同。

我要做的改变：用“短反馈验证 + 全链路门禁 + AAR 反哺”抵消生成速度带来的风险。
