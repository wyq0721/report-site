仓颉（Cangjie）代码生成：错误分析与常见错误清单

面对训练数据稀少的新语言，如何把"模型凭记忆写"升级为"拦截层 + 编译闭环"

整理日期：2026-06-23  ·  工作目录：apilevel_ws/wikishare  ·  适用：用 AI 生成仓颉业务逻辑
核心结论： 编译器报错已经足够准确，优化报错信息是低杠杆。真正的杠杆在两处—— ① 减少这些错被"生产"出来的概率（写之前拦截）； ② 让 agent 自己消化编译器反馈（自动 cjc 编译→改），而不是把报错丢给人。 更关键的洞察：碰到的错不是一类问题，需要相反的解法——见下表。

一、两个错误家族（解法相反）

A 类：系统性 / 可预测 规则可消除 B 类：上下文相关 需检索
根因 训练先验太强，模型反复犯同一个错（如把别的语言习惯带过来） 不知道"这个具体签名 / 这个包导出了什么"
典型 三目 ?:、Option / match 惯用法、分号习惯 API 参数类型、泛型约束、包成员对应
解法 静态规则前置注入（checklist / lint / memory），接近 100% 拦截，根本走不到编译器 写之前检索真实签名 / 导出（codegraph / 文档 / 读现有 .cj），猜不得
能否靠规则消除 ✅ 能，一条规则永久解决 ❌ 不能，靠检索或编译闭环兜

二、当前碰到的三类问题归类

问题归属对策
1. API 调用参数或类型给错 B 类 写前查真实签名（codegraph / 文档 / 跳转定义），不凭记忆
2. Option 类型、match 语法使用错误 A 类（多数） 把正确惯用法写进规则清单前置注入
3. 包名导入跟实际成员对应不上 B 类 对照文档 + 现有 import + cjc，查真实导出

三、两个实例分析（一边一个家族）

实例 1 — 三目运算符 A 类 · 纯系统性

报错：expected '.', '(', '[', '{' or '?' after '?' / expected operator or end of expression, found ':'

仓颉没有 C 风格三目运算符 ?:。模型写 cond ? a : b 是 C / JS / ArkTS 的肌肉记忆。正确写法是 if-else 当表达式用：

❌ 错误（迁移惯性）
.fontWeight(this.currentTab == index
    ? FontWeight.Bold
    : FontWeight.Normal)
✅ 正确（if-else 是表达式）
.fontWeight(if (this.currentTab == index) {
    FontWeight.Bold
} else {
    FontWeight.Normal
})

→ 这种错会一犯再犯，一条规则就能永久消除，价值最高。

实例 2 — 泛型 Some(u) 类型不匹配 B 类 + 类型系统

报错：return type 'Option<Generics-T>' should match expected type 'Option<Class-User>'；note：constraint 'Generics-T <: Class-User'

根因是仓颉泛型不变（invariant）：即使 T <: User，Option<T> 也不等于 Option<User>。所以在声明返回 Option<User> 的函数里 return Some(u)（其类型是 Option<T>）无法匹配。

❌ 错误
// 返回类型声明为 Option<User>
// u: T, 约束 T <: User
return Some(u)   // 实际是 Option<T>
✅ 方向（二选一，看函数语义）
// 方案A：显式上转
return Some<User>(u)

// 方案B：返回类型改成 Option<T>

⚠️ 具体选哪个要看完整函数签名，应以 cjc 为准。这类错靠静态规则防不住，只能靠类型推理或编译闭环兜。

四、改进方向（杠杆点）

  • 把 A 类沉淀成"仓颉反模式 checklist"，写仓颉前先注入。每踩一个新的系统性坑就追加一条 → 复发率单调下降。（仓库已有 checklist 文化 + cangjie-regulations skill）
  • 把编译闭环自动化：让 agent 自己 cjc 编译 → 读报错 → 改，而不是把报错贴给人。这样"编译器很准"的优点被 agent 吃掉，人看不到这些 round-trip。
  • B 类无法靠规则：写之前强制检索真实签名 / 包导出（codegraph、读现有 import、cangjie-std/stdx 文档）。
建议工作流
读清单 + 检索真实签名→ 写一小段→ 自动 cjc 编译→ 失败按报错改→ 新坑追加进清单

五、降低"反复迭代"的消耗

现状：agent 已经能自动消化编译错误（杠杆②已落地）。但实测改不到一次到位——一轮改完冒出新错、来回震荡，迭代本身在烧时间/构建/token。 把消耗拆成可攻击的三个因子： 
迭代总消耗  ≈  每轮残留错误数  ×  迭代轮数  ×  单轮构建成本

为什么收敛不到一次到位

迭代变多的原因机理攻击因子
级联报错 语法错会掩盖下游语义错；改完语法，新一批错才暴露。早期错误数"不可信"。 轮数
改一处、漏多处 编译器只指 1 处，但同类反模式（如多个三目）在文件里有 N 处 → 逐个改 = N 轮。 残留错误数
改症状不改根因（震荡） 在新语言上"猜着改"，改了反而引入新错，来回摆动。这是最贵的黑洞——每次猜成功率低。 轮数
全量构建慢 每轮跑全量 build，即使轮数不多，墙钟时间也被放大。 单轮成本

五个对策（按收益排序）

对策做法
① 按"类"改，不按"行"改 最大收益 报错指 1 个三目，先全文件 / 全仓库搜同类反模式，一次全改。把 N 轮压成 1 轮。直接砍"残留错误数"。
② 先清语法错，再动语义 语法错会掩盖真实错误数；语法没干净前，不要信任错误列表、不要改语义。避免无效轮次。
③ 根因优先，拒绝震荡 B 类错（签名/类型）禁止试错改。一次改不对就停手，去查真实签名 / 文档 / 跳定义，确认后再改。新语言上"少猜一次"比"多试三次"省。
④ 内外两层循环 内层用最快反馈（LSP 诊断 / cjc 仅检查 / 增量构建）攒一批有把握的改动，外层才跑全量 build。直接砍"单轮成本"。（有 build-cangjie-...-incremental 技能）
⑤ 每轮消耗转成规则（摊销） 凡为某"类"错烧过的迭代，写进下方清单 E? → 下次该类成本归零。迭代成本每类只付一次。
两层循环示意
批量改→ 内层：LSP / cjc --check / 增量（快）↻ 攒一批 → 外层：全量 build（慢，少跑）→ 新坑写进 E?（摊销）

本质：迭代消耗应"每个错误类只付一次，然后摊销到 0"。这正是清单存在的经济学意义——把今天烧掉的迭代，沉淀成明天的零成本，连通"前置拦截①"与"自动消化②"。

六、仓颉常见错误清单（可增长）

📄 仓颉常见错误.md

用法：写仓颉前先读本清单（A 类规则前置拦截）。 标签 已确认 来自实测报错；原则 是通用方法；待核实 需用 cjc / cangjie-std 文档确认后再固化。

E1 · 不要用三目运算符 ?: 已确认 A 类
仓颉没有 C 风格三目；if-else 本身是表达式，直接当值用。
❌  cond ? a : b
✅  if (cond) { a } else { b }
触发场景：从 ArkTS / JS / C++ / Java 迁移惯性。出现在任何需要"条件取值"的地方（属性赋值、链式调用参数等）。
E2 · 泛型不变：Option<T> ≠ Option<User> 已确认 B 类
仓颉泛型 invariant。T <: User 不代表 Option<T> 可赋给 Option<User>。
❌  // 返回类型 Option<User>，u: T (T <: User)
    return Some(u)          // 实为 Option<T>
✅  return Some<User>(u)   // 显式上转
✅  // 或把返回类型声明改成 Option<T>
取舍：看函数语义，以 cjc 为准。规则防不住，靠类型推理 / 编译闭环。
E3 · Option 解构用 match 待核实 A 类
处理 Option 用模式匹配，别假设有 null / 直接解引用。常见形态如下，请用 cjc 确认后固化。
match (opt) {
    case Some(v) => useValue(v)
    case None    => handleEmpty()
}
提醒：match 各分支类型要一致；穷尽所有构造子（Some / None）。
E4 · 包导入必须对应真实导出成员 原则 B 类
不凭记忆 import 包或成员——包路径、成员名都容易幻觉。
❌  凭印象写 import；猜某函数在某包里
✅  ① 抄仓库现有 .cj 的 import 写法
✅  ② 对照 cangjie-std / cangjie-stdx 文档列出的真实包
✅  ③ codegraph / grep 确认符号确实存在、在哪个包
✅  ④ cjc 编译，导错立即报错
顺序：检索优先级 现有代码 ≈ 文档 > 记忆（记忆仅辅助）。
E5 · API 参数 / 类型以真实签名为准 原则 B 类
参数个数、顺序、类型、是否带默认值——凭记忆容易错（尤其重载、泛型约束）。
✅  调用前先跳定义 / 查文档拿到真实签名，再写调用
✅  泛型调用注意约束（如 T <: SomeType）与显式类型实参
关联：与 E2 同源——类型系统相关错都应"先确认签名再写"。
E? · 新坑追加模板 待补充
每次踩到新的系统性坑，按此结构追加一条，让清单单调增长。
E? · <一句话标题>            [已确认 / 原则 / 待核实] [A/B 类]
why : <根因，为什么模型会犯>
❌   <错误写法>
✅   <正确写法>
when: <触发场景 / 提醒>
