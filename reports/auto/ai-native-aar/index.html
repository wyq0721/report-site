03问题 → 根因 → 改进措施

问题（案例锚点）根因（Agent 行为模式）改进措施
整数形式先实现后废弃（A1）；包归属猜错、开发期一度破坏源兼容（A2，合入前修正） 先验压过规格，缺口补以猜测。已写明的弱表述约束输给训练先验（A1）；spec 没覆盖的外部事实被自信补齐（A2）——两案例中 Agent 均未表达过"不确定"。 spec 前置。硬约束逐条转成负向验收用例（"整数形式必须报错"）——文字约束拦不住先验，验收用例拦得住；外部接口逐项标注「已求证（出处）/ 假设」，假设项先求证再动手。
无关 weak-symbol "修复"整链混进特性 PR，追问后删除（B1） 改动范围自行蔓延。生成跨模块代码的成本近乎为零，遇到挡路报错倾向就地修通，而不是先问"这是不是本特性该修的问题"。 范围钉死。无关报错单独开 issue 记录，禁止在特性 PR 里顺手修；跨模块/平台方案先用一段话对齐方向，认可后再动手。
!1417 修 ICE 埋下性能炸弹，潜伏 48 天由用户报出（#927） 不评估"旧 bug 被谁依赖"。Agent 把 bug 当待消除的错误，不会问"错误行为是否已成为其他路径的隐形支柱"（Hyrum 定律）。 修复 PR 模板加一行必填："旧行为是否可能被依赖？修复后哪些路径首次真实生效？"由作者作答、由独立审计核验（审计三问之一）——填了不等于过，防止落回自证闭环。
!1815 验证 335 用例逐字节一致，仍漏 O2 回归穿透 main 自证闭环：点验证强、面验证弱。修复、回归用例、验证清单同出作者上下文，同一盲区穿透三者；上下文里没有"优化档会改变推断行为"这个假设，清单再长也不会出现 O2——指望作者 Agent 独立检出自己的盲区，结构上不成立。 验证矩阵进机检。门禁对 Sema 相关变更增跑 O2 档（按变更路径触发，控制 CI 成本），本地最少「默认 + O2」两档；语义收窄类改动（过滤/裁剪/短路）必附同机前后性能对比。固化进 CI，不依赖人记得。
三级修复全链仅 1 条实质检视意见；评论数门禁两次由非实质评论满足 异质视角稀缺，且指标不度量它。两案例所有有效拦截都来自作者之外（weak-symbol 追问、genArity、后冒烟）——第二视角有效但供给不足；门禁只数评论条数、不辨信息量，缺陷首先在指标设计，而非某次操作。 独立 agent 审计（试点）。不共享作者上下文的 agent 固定回答三问：旧行为被谁依赖？验证矩阵缺哪档？语义收窄波及哪些场景？分歧由 reviewer 仲裁。先只对 fix-of-fix 场景试点，度量拦截率/误报率，达标再扩面替代评论数指标；审计 agent 同样有先验盲区，兜底仍是机检矩阵与后冒烟。
规范意见占近四成；覆盖率欠账；#927 未关闭 模板类错误反复出现 + 质量项无强制闭环点。规范错消耗人的评审带宽；覆盖率核算在合入后才做、80% 只是规范要求而非强制门禁——规则存在但没有卡点。 机器拦规范、闭环有终点。版权 / format / 头文件实现 / CODEOWNERS 进 pre-PR hook；覆盖率核算前置为合入条件（补上缺失的卡点）；修复链完结回归验证并关闭 issue（因果链留在 issue 内，!1815 的做法固化为惯例）。

04值得保持的

  • 根因分析质量：三级修复的根因（checkpoint 深度、欠定候选、结果位约束）全部准确，PR 描述完整交代了与上一级修复的因果关系。
  • 测试看护有设计：!1791 的 8 个 LLT 用例带 3 倍裕量的 TIMEOUT 设计，慢 CI 不误报、真回归必超时。
  • 后冒烟有效：O2 回归在合入当晚即被抓住，止损窗口不到一天。
  • 过程留痕：备份分支零丢失、检视账本逐条闭环、commit message 引用 note id 可追溯。

一句话收尾：Agent 能把"怎么修"做对；"该不该这样修、修完还会动谁"必须由不共享作者假设的第二视角来守。两个案例里所有穿透到用户的错误，都发生在只有作者一个视角的时候。
