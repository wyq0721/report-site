2. 期望发生什么

对一个剪枝型性能修复来说，期望不是“跑赢既有用例”，而是：被剪掉的候选必须确实永远解不出来； 任何“这个候选可以安全丢弃”的判断，都要穷尽它可能被约束的所有通道，并保证不会误伤任何一个合法程序。
期望项具体标准为什么重要
剪枝正确性 “候选无法确定”的判定必须覆盖所有约束来源：成员参数、成员返回值、以及外层上下文（二元运算、is、as 等）。 排除式剪枝一旦把可解候选误判为不可解，就会把合法程序改成推断失败。
语法形态覆盖 lambda 参数最常见的使用形态（含下标 a[i]）都应被“成员使用”枚举识别。 枚举不全会让整类用法在剪枝里“隐身”，既测不到也保护不到。
行为不变性 “当前能编译的程序行为不变”需要针对剪枝盲区构造反例来证明，而不是只看既有语料 diff。 既有语料不覆盖盲区时，“诊断逐字节一致”只是虚假的安全感。
失败兜底 当剪枝把候选集清空、但仍存在外层判别器时，应回退到完整候选集，绝不让外层无候选可用。 剪枝的本意是去掉“纯噪声”候选，不应在还有判别器时把推断逼到死路。

3. 实际发生了什么

无标注 lambda 参数靠“它在 lambda 体里怎么被使用”来反推类型。歧义候选会被塞进类型变量的 sum 约束 （AddSumByCtor），并为泛型实参分配占位变量。!1815 的核心是排除式过滤 CanDetermineCandidate：

如果某个泛型候选的泛型参数不能全部由所匹配成员的“参数类型”确定，就不进 sum。 例如 HashMap<K, V> 经 operator [](key: K): V 匹配时 V 只在返回位， 注入的占位 V 永远无解，拖着所有依赖约束“落不定”，触发字面量重试的组合式重算。过滤它，耗时回到 0.11s。

问题在于这个“参数才算数”的覆盖模型漏掉了一整类：返回位泛型可以被外层上下文约束。冒烟用例：

// lambda00920.cj · LEVEL 2 / RUN-EXEC（需编译并运行）
main(): Int64 {
    let f = {
        a => a[0] == 1
    }
    if (f is (Array<Int64>) -> Bool) {
        return 0
    }
    return 1
}

a[0] 是 Array<T>.operator [](Int): T，T 只在返回位。 而 T 其实完全可由外层确定：== 1 把结果约束成可与 Int64 比较， 外层 is (Array<Int64>) -> Bool 更是直接钉死。!1815 只看参数类型（Int，无泛型）， 判定 T 落不定，把 Array<T> 误删，推断失败、用例回归。

缺陷 A 下标根本没被识别

!1815 的扫描器只处理 a.f(...)（CallExpr）和 a.f（MemberAccess）， a[i] 这种极常见写法压根没产出 MemSig。

缺陷 B 只认参数覆盖

覆盖模型只统计成员参数类型里的泛型，忽略了“返回位泛型可被外层结果约束确定”这一通道。

4. 根因分析

两个缺陷叠加，但根因是同一个：!1815 把“候选解不出来”的判定建立在了一个过窄的不变量上—— “泛型只能由成员参数确定”。这个不变量只对参数通道成立，却被当成了全集，于是返回位泛型（以及它背后的外层约束源） 被整体漏掉。PR 描述里那句“被排除的候选反正解不出来，所以不改变任何当前能编译程序的行为”，正是这个过窄不变量的直接产物，事后被 lambda00920.cj 证伪。

层次根因
直接原因 覆盖检查只数成员参数类型里的泛型；返回位泛型被判为不可确定 → 合法候选 Array<T> 被剪掉。
实现缺口 “成员使用”枚举不含 SubscriptExpr，下标用法在整条链路里缺席。
深层原因 排除式优化默认“我已枚举完所有约束来源”，缺少“还有我没想到的约束通道”的防御性假设和兜底。

5. !1832 怎么补的

思路：把“覆盖”从“仅参数”扩展为“参数 ∪（在有外层结果约束时）返回值”，并加一道永不清空候选集的兜底。

修法实现
识别下标 新增 FindCandidatesFromSubscript，把 a[i] 记成 MemSig{"[]", false, indexCount, 0}。
标记结果约束 扫描器进入 BinaryExpr / IsExpr / AsExpr 的左侧时置 resultConstrained=true，相应成员使用记入 resultConstrainedMemSigs。
结果覆盖 CollectMemberResultCoverage 对结果受约束的成员，把其返回类型里的 owner 泛型也算作可确定，于是 Array<T>.[]→T 的 T 被覆盖、候选得以保留。
清空兜底 过滤后若 sumTys 为空且存在结果约束，则恢复全部候选 sumTys = declTys——外层结果约束可能是唯一判别器，不能因证不出就清空。
快路径收口 把原来的 uniq 快路径用 resultConstrainedMemSigs.empty() 网住，保证有结果约束时走完整路径。
关键设计：只在存在结果约束时放宽。无结果约束的路径完全保持 !1815 的过滤逻辑， 所以 927 的性能优化不回退——这是一次“收窄修复范围、不动已验证收益”的补丁。

6. 为什么自验证没拦住

!1815 的验证是 335 个 Sema LLT 诊断逐字节一致 + !1418 复现用例 + std 全量重编。但触发回归的用例不在这张网里：

  • 套件错位：lambda00920.cj 在 HLT 套件，且是 LEVEL 2 / RUN-EXEC（要真正编译并运行），不在那 335 个 Sema 诊断集中。
  • 盲区未覆盖：验证集里没有“lambda 参数下标访问 + 返回值被外层约束”这个组合。
  • 对比方法的局限：“诊断逐字节一致”只覆盖被测样本；对排除式改动，真正该问的是“被排除集合里有没有合法程序”，这需要针对盲区构造反例，而不是跑既有语料看 diff。
缺口不在回归对比方法本身，而在验证语料的覆盖面：不变的是样本，而剪枝的盲区恰好不在样本里。

7. 经验与改进

经验下一步改进优先级
“反正解不出来”是危险的优化前提——剪枝必须穷尽所有约束通道（参数 / 返回值 / 外层），并配“永不清空候选集”的 fail-safe。 把“枚举所有约束来源 + 兜底回退”作为剪枝类改动的设计自检项，评审清单固化。 P0
“诊断逐字节一致”≠“行为不变”。 对排除式改动，要求作者针对盲区构造反例用例（返回位泛型 / 外层约束），并随修复一起入库。 P0
常见语法形态（下标 a[i]）被整条链路漏掉。 实现“成员使用”枚举时对照语法形态清单（调用 / 访问 / 下标 / 操作符 / is / as）逐项确认。 P1
合入后冒烟（含 HLT + 运行）兜住了，但代价是已进入 main。 对性能剪枝型修复，在合入门禁加一组“剪枝盲区”定向用例，把验证从“既有语料 diff”前移到“针对性反例”。 P1
