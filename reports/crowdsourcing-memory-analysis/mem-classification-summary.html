DWARF 破解"无栈"难题:fp_unwind 下 58%(129MB)分配无调用栈 —— 因 FFRT(HarmonyOS 协程线程池)用纤程栈,帧指针展开断在纤程边界。改用 DWARF(.eh_frame)展开后无栈率 58%→3%,这 129MB 得以归因(主体 = FFRT worker 上的本地 DB / 资源解码 / 引擎初始化)。

第四层 · 仓颉 GC 堆锯齿 + modal 收益实测④

峰−谷振幅 ≈ 110MB 几乎全为瞬时对象;modal 可填平其中 ~30% = 26–32MB
仓颉 GC 堆是独立锯齿([anon:cangjie_heap],非 malloc,不在第二层)。非分代 copying GC 按堆增长阈值触发:峰值恒定 ≈145MB(阈值)、存活地板 ≈35MB(真实 live)。存量态空闲实测 36MB = 正好落在地板,印证GC 把它死死压住、不随会话累积上涨。
GC 频率 ∝ 刷新频率:3s→~13s、1s→~6s、0.5s→~4s 触发一次。峰−谷几乎全是瞬时 JSON/DataModel —— 正是 modal/@local region 化的作用空间。
modal 预估削峰 = (峰−谷振幅) × 可 region 化比例(JSON/DataModel ~30%)≈ 26–32MB ≈ 应用峰值 PSS 的 2–3%,并降低 GC 频率。锯齿等额传导到整机峰值 RSS(Δ 相关 r≈0.6–0.8)→ 削峰 = 抗 LMK(Low Memory Killer)。

会话累积的 +330MB 几乎全在 jemalloc / ArkTS 堆 / ArkWeb;仓颉 GC 堆被 GC 压在 ~35MB 恒定。所以仓颉相对占比随使用不升反降(8%→3%)。结论:modal 的价值不在"降存量基线",而在"削刷新瞬时峰"。

结论 · 哪块最值得投入

优先级目标依据 / 手段
🥇 最高native heap(jemalloc)38% — JS/UI 引擎 + 启动留存最大且随会话涨;持有大头是 FFRT 后台工作(本地 DB/资源解码/引擎初始化)与 ArkTS/ArkUI。手段:DWARF 已就位,可按符号下钻 ArkUI 留存与启动 SQLite/资源。
🥈 高ArkTS 引擎堆 + 框架 ≈ 13%原生 UI/JS 对象、组件树;刷新留存最大头(10MB/轮)。查长列表/未释放 @State/图片缓存。
🥉 中RN(JS 堆 6% + 原生在 jemalloc)/ ArkWeb 5%RN 在本(仓颉/ArkUI)订单页 native 很小(churn 9MB/留存 0.6MB);真实占用需切到 RN 页面再测。
➖ 辅助Cangjie / modal(GC 堆 ~3%)占比小、削峰 2–3%。价值在削峰抗 LMK + 降 GC 频率,非降总量。整机大盘 modal 不是主战场。

口径与局限

  • native heap 不可由 smaps 细分(jemalloc 单一池),归因靠 malloc-hook;实测②为新启动态(比例代表存量,绝对值偏小),实测③为刷新窗口(不含基线)。
  • GPU(GL+Graph)不在 smaps,随渲染状态波动(58MB 为空闲态),计入 hidumper Total。
  • 仓颉 GC 堆是命名匿名映射不走 malloc,故不在第二/三层;modal 仅作用于此 ~36MB 中可 region 化的 JSON/DataModel(~30%)。
  • RN 真实占用未完全测出:本订单页是仓颉/ArkUI 页,RN 在 native 极小;要量 RN 需在采集时导航到真正的 RN 页面。
  • 归因走 last_lib_id(直接调用 .so);DWARF 已含完整调用链,可进一步按符号下钻。
