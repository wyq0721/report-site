读"谷→峰"前,先看口径(否则会误读):峰 ≈ GC 前高水位、谷 ≈ GC 后存活地板,均为 6s 采样近似(真实峰更高、谷更低)。不要跨场景直接比峰值:前台峰(149)反而低于锁屏峰(177),并非前台更省 —— 而是前台分配快、GC 频繁把堆压住;锁屏分配慢、长时间不触发 GC,垃圾堆成陈旧高水位(177)直到锁屏 trim。两场景存活地板相近(43 / 35MB),差异在 GC 触发节奏,不在真实用量。
modal 收益(两类,分清楚):
① 内存收益(削峰):峰−谷(107MB 前台 / 142MB 锁屏)几乎全是瞬时;其中 ~30% 为可 region 化的 JSON/DataModel(C1+C2,见 mem_report.html 拆解)→ 每周期可压 ~23MB,AnonPage/PSS 同步 → 降整机峰值 RSS,对抗 LMK。
② CPU 收益(降 GC):前台每 ~18s 一次 GC 由瞬时 churn 触发;region 化后这些对象不进 GC 堆 → GC 次数与拷贝开销下降(jank 收益)。
③ 空闲不囤垃圾:锁屏 177MB 高水位大部分是没被回收的瞬时垃圾;modal 在作用域退出即释放,空闲不会坐在这堆垃圾上。
