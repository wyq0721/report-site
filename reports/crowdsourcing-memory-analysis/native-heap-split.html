结论

① 最大块是 FFRT worker pool ≈ 101 MB / 45% —— 无栈。HarmonyOS 的 FFRT(Function Flow Runtime)用协程/纤程栈跑并发任务,fp_unwind 在纤程切换边界断掉,记录不到调用栈。这块是 ArkTS 并发/taskpool、V8 后台编译&GC、图片/媒体解码、网络等后台任务共用线程池的分配(其中 74 个大 buffer 合计 ~102MB)。要再拆只能用 DWARF(.eh_frame)展开重抓一遍。
② 最大可归因属主是 ArkTS/ArkUI 引擎 ≈ 48 MB / 22%。HarmonyOS 原生 UI/JS 运行时(libark_jsruntime + libace_compatible + libace_napi)。订单列表是 Cangjie/ArkUI 页,native 主要由 ArkTS/ArkUI 撑起。
③ RN ≈ 9 MB、Cangjie ≈ 11 MB —— 在本页 native 里都很小。RN 原生(react_render/mrn/aki)+ V8 合计不到 10MB;Cangjie C++ 运行时 ~11MB(仓颉 GC 堆另在 [anon:cangjie_heap],不在 jemalloc)。与既有结论一致:native 大盘不是 RN / 不是 Cangjie。

口径与局限

  • 新启动态 222MB,非存量态 392MB。属主构成比例可代表存量态(同样的属主放大),但绝对值不同;要拿存量态精确拆分,需重启→重放点击负载→DWARF 全量(代价:重新人脸上线、trace 更大)。
  • 无栈 58% 的根因是 fp_unwind + FFRT 协程栈,不是数据缺失。DWARF 展开能拿到任务自身的帧(纤程边界以上),可把这 129MB 大部分归到真实属主。
  • 仓颉 GC 堆不在此表:它是 [anon:cangjie_heap] 命名匿名映射(AnonPage),不走 malloc。本表只含 jemalloc/brk 等 C/C++ malloc。
  • 已归因走 last_lib_id(直接调用方 .so),非完整调用链根因;同一 .so 可能被多上层调用。
