哪块最值得投入(结论)

① native heap(jemalloc)≈ 392 MB / 38% —— 最大头,但 smaps 无法细分。它是所有 C/C++ malloc 的混合池:RN 原生(folly/jsi/react_render C++、V8 堆外)、Cangjie C++ 运行时、图片/解码缓冲、系统库都混在里面(jemalloc Free 仅 ~38MB,说明基本是真实占用,非保留)。要知道"谁占的",必须做原生堆剖析(DevEco Profiler 的 Native Memory / malloc hook / hiprofiler nativehook,按调用栈归因)。这是投入产出最高的下一步。
② ArkTS 引擎堆 ≈ 137 MB / 13% —— 第二大。HarmonyOS 原生 ArkUI/ArkTS 的 JS 堆。若 App 有大量 ArkUI 原生页面/长列表/未释放的 @State,这里有空间(JS 对象、组件树、图片缓存)。
③ RN(React Native)≈ 直接可归因 84 MB / 8%,真实更大。JS 堆(66MB,JSVM_V8_HEAP_*_RNOH_JS)可见;但 RN 的原生 C++(libreact_render*/folly/jsi/V8 堆外)都 malloc 进了上面的 jemalloc 池,smaps 看不到。RN 的真实总占用要靠①的原生剖析才能切出——很可能是最大的"产品可归因"消耗。
④ Cangjie ≈ 90 MB / 9% —— 占比小,modal 天花板有限。与此前结论一致:modal 削峰约 2–5% 应用内存。若目标是整机内存大盘,Cangjie/modal 不是主战场;主战场是 native heap 与 ArkTS 堆。modal 的价值在削峰(抗 LMK)与降 GC,而非降总量。

口径与边界

  • native heap 不可由 smaps 细分:jemalloc 是单一池,RN/Cangjie/系统的 malloc 混在一起。归因需 malloc 级原生剖析。
  • GPU 内存(GL+Graph ≈ 58MB)不在 smaps:hidumper Total(1127MB)≈ smaps(1037MB)+ GPU + 采样漂移。本报告子系统基于 smaps(可归因常驻集)。
  • 交叉校验:hidumper native heap=393MB / ark ts heap=112MB,与 smaps 归因一致。
  • .so 为文件页 Pss(共享按比例),代码段占比天然偏小;subsystem 的 .so 行只反映代码/只读数据,运行时数据在各自堆/jemalloc。
