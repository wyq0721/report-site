③ 手动GC A/B(仓颉堆削峰)

手动GC A/B 证明了削峰空间是真实、可回收的:刷新前一次手动GC 就把仓颉堆峰值从 95 压到 29MB(−69%)、应用 PSS 峰值 581→491MB。说明刷新态的"峰−谷"几乎全是瞬时可回收对象——正是 modal/@local region 化要拿掉的部分。
但手动GC 不是免费的:它是全堆 STW copying GC,每次刷新都强制一次 → 削峰以 CPU 开销 + 潜在卡顿(STW)+ 每字节搬移成本为代价。modal/@local 的价值正在于:用 region 化在退出作用域时 O(1) 丢弃瞬时对象,拿到同样削峰而不付全堆 GC/STW 代价。

④ 多页面浏览后 · 整体内存抬升,但仓颉堆不受影响

同一进程(新构建 pid 63397)手动多点击浏览若干页面后:应用整体 PSS 从 ~472MB 抬升到 ~706MB(+234MB),且离开页面回到订单列表后仍常驻。但仓颉 GC 堆纹丝不动:多页面后空闲 19–20MB、下拉刷新 21–25MB,与浏览前(~21MB)完全一致。
抬升的 +234MB 全在仓颉堆之外:主要是 ArkWeb 内嵌浏览器引擎(libarkweb_engine.so ~71MB,首次打开网页/H5 懒加载后常驻)+ native/ArkTS/会话缓存的累积(详见 rn_state_*/mem_breakdown_rn.html)。结论:多点击页面推高的是"整机内存大盘",不是仓颉堆——仓颉/modal 对这类增长无能为力,也不背这个锅。整机内存主战场是 ArkWeb / native / ArkTS,与既有子系统归因一致。

⑤ 结论

三件事各归各:① 下线静置=仓颉堆平线 ~19MB 零 GC,波动全由刷新驱动(旧构建峰值 95MB,低于上线 ~145MB)。② 手动GC/modal 能把刷新锯齿削平(峰值 95→29MB),作用域=仓颉堆内的瞬时对象。③ 多点击页面推高整机 PSS(+234MB),但那是 ArkWeb/native/ArkTS,与仓颉堆无关、modal 削不到。⇒ modal 优化仓颉堆锯齿削峰;整机内存大盘增长要靠 ArkWeb/native/ArkTS 侧治理。两者不可互相替代。
