③ modal 落地点 (按收益排序)

1JSON 解析 → DataModel 瞬时树 (最大、最干净)
stdx.encoding.jsonstdx.serialization~15.5 MB
  • 证据:DataModelStruct 8,879 · JsonObject 9,768 · HashMap<String,JsonValue> 9,769 · Field 55,244 —— 整套指针密集嵌套树。
  • 对 DataModelStruct 做 incoming-reference:绝大多数树顶层无任何外部引用,唯一 retainer 是树内自环 (Field→RawArray→ArrayList→自己)。--show-thread 显示所有线程都 parked、无一帧在解析中。
  • => 这片是已死、GC 尚未回收的瞬时树。在非分代 + copying collector 下,存活到 GC 时刻者要按字节 evacuation memcpy,瞬时分配并不免费。
  • 收益:region 化 parse→遍历→抽取→exclave 小结果,~15.5 MB 结构 + 其 Byte 底层从 GC 托管堆移除,并消掉每轮 GC 对深图的 trace+evacuate。
2JsonParser 钉住 输入缓冲 + 解析树 (设计性泄漏)
7,849 个存活 JsonParserretained 5.87 MB
  • outgoing-reference:单个 JsonParser 同时持有 RawArray<JsonValue>(输出树)与 ArrayList<UInt8>/RawArray<Byte>(输入字节缓冲)。
  • => 只要一个 parser 引用被(哪怕间接)存住,整棵 JSON 树 + 源文本就跟着不死。
  • 收益:在 region 内构造 parser,仅抽取后的小值经 exclave 逃逸,parser 与输入缓冲随 region 销毁 —— 从语言层消除 “parser 寿命 = 树寿命” 的隐患。
3监控/上报 bean 构造路径 (部分契合)
cjjson::AnyType ~9.6 MBDaBaiBean 1,352
  • cjjson::AnyType 家族主锚定者是 com_meituan_banma_nativecore.monitor.bean::DaBaiBean(5,500+ 条引用)与 ohos.ark_interop::JSObject/JSValue(JS 互操作桥)。
  • 会逃逸的部分:bean 进了上报队列 (LinkedBlockingQueue<Message>) —— modal 不能回收,属 “flush 速率 < 生产速率” 的积压,另案处理。
  • 可 region 化的部分:构造 bean = “解析整个入参 JSON map → 只抽几个字段”,中间 AnyType map 会死、只有小 bean 逃逸 —— modal 收编这个中间体。

④ 收益空间 (量级,非假精度)

可直接 region 化 / 净移出 GC 堆
① JSON/DataModel 瞬时图 ≈ 15.5 MB 结构 + 其 Byte 底层大部分,叠加 ② parser 输入缓冲。 保守看 全堆约 1/4 ~ 1/3 (~15–20 MB) 是 region-scopable 的纯瞬时。
额外收益
消掉这片深指针图每轮 GC 的 trace+copy → 降低 GC 频率/停顿; 并从语言层堵死 ②/③ 的 “瞬时对象被意外存活” 泄漏面(本 dump 出自 memory_leak 可靠性路径,本身即堆积现场)。
modal 不解决
真正逃逸进上报队列的 DaBaiBean payload —— flush 速率问题,需另案处理 (扩大 flush 频率 / 背压 / 采样)。
实施顺序
需先有 Tier-0 (String/Array/StringBuilder) 的 @local! 构造器, 再到集合 (ArrayList/HashMap),JSON 树才能整棵进 region。改造靶点落在 stdx.encoding.json(JsonParser/解析入口)与 stdx.serialization(DataModel.fromJson 的 buildDMStruct/buildDMSeq 递归)。
