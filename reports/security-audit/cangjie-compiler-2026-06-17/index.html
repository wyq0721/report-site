确认发现 (Confirmed Findings)

[VUL-001] 未捕获 std::sto* 异常导致编译器崩溃 Medium CONFIRMED

  • 类别：Parser/Deserializer DoS (COMP-PARSE-002 / COMP-NUM-001)
  • 位置：src/ConditionalCompilation/ConditionalCompilation.cpp:167-169（经 :431 触达）；src/Sema/NativeFFI/Java/AfterTypeCheck/InteropLibBridge.cpp:1758
  • 状态：历史 MED-13 的不完整修复 —— 安全包装器 StdUtils::Stoul/Stoi（返回 std::optional，内部 try/catch）已存在，但这两处仍直接调用原始 std::stoul/std::stoi。

证据 (Source → Path → Sink)

实例 A —— 条件编译版本比较（源码可控）

// ConditionalCompilation.cpp:430-437
if (conditionStr == CJC_VERSION_STR) {
    VersionInfo rightVersion = ParseVersion(right->stringValue);   // right->stringValue 来自源码字符串字面量
    ...
}
// ParseVersion: 167-169
std::regex pattern("([0-9]+)[.]([0-9]+)[.]([0-9]+)");   // 数字组无长度上限
...
info.major = static_cast<uint32_t>(std::stoul(match[REGEX_MAJOR_IDX].str()));  // 无 try/catch
  • Source：.cj 源文件中的条件编译条件 @When[cjc_version >= "<version>"]，RHS 是字符串字面量（:409 已校验为 STRING）。
  • Path：CompileStrategy::ConditionCompile → HandleConditionalCompilation → EvalJudgeBinaryExpr → ParseVersion。属于正常前端编译流程。
  • Sink：std::stoul("99999999999999999999")（20 位 > ULONG_MAX）抛出 std::out_of_range。范围校验 IsValidVersionNumber 在 stoul 之后，无法阻止异常。

实例 B —— Java 互操作库版本（依赖可控）

// InteropLibBridge.cpp:1750-1758
if (... versionDecl->initializer->TyKind() != TypeKind::TYPE_INT64) { ... }  // 类型为 Int64
auto libVersion = std::stoi(libVersionLit->stringValue);   // 用 stoi(int!) 解析 Int64 字面量

字面量类型是 Int64，却用 std::stoi（int，上限 ~2.1e9）解析。任何合法 Int64 版本值 > INT_MAX（如 3000000000）即抛 std::out_of_range。Source = 被处理的 Java interop 库包中的 VERSION 声明。

影响

main.cpp:104 顶层只 catch (NullPointerException)，不捕获 std::out_of_range/std::exception。异常逃逸 → std::terminate() → abort() → SIGABRT（在 SignalUnix.cpp:24 的崩溃信号列表中）→ 以 ICE 形式硬崩溃退出。即：一段构造的源码或一个恶意/畸形依赖即可可靠地崩溃编译器（CI、包构建、IDE 后台编译、在线 playground 等场景的 DoS）。

复现 (Reproducer)

// crash_condcomp.cj —— 触发实例 A
@When[cjc_version >= "999999999999999999999999.0.0"]
func f(): Unit {}

main() {}

预期：cjc crash_condcomp.cj 抛出未捕获 std::out_of_range，以 SIGABRT/ICE 终止，而非输出 “invalid cjc_version format” 诊断。（实例 B 需构造 VERSION: Int64 = 3000000000 的 interop 库，机制相同。）

修复

auto v = StdUtils::Stoul(match[REGEX_MAJOR_IDX].str());
if (!v) { /* 报 conditional_compilation_not_support_cjc_version_format 诊断 */ return info; }
info.major = static_cast<uint32_t>(*v);

实例 B 用 StdUtils::Stoll（匹配 Int64）并对 nullopt/越界发诊断。建议全仓禁用裸 std::sto*（可加 lint 规则）。

[VUL-002] SIGINT 处理器调用非异步信号安全的 DeleteTempFiles() Low CONFIRMED

  • 类别：Signal/Async Safety (COMP-SIG-001)
  • 位置：src/Utils/SignalUnix.cpp:76-80
  • 状态：崩溃信号路径已修复（ConcurrentSynchronousSignalHandler → DeleteTempFilesSignalSafe()，仅 unlink/rmdir），但 SIGINT（Ctrl-C）处理器未同步修复，且该处理器在 release 与 debug 构建中均编译（位于 #if RELEASE/#else/#endif 之外）。

证据

// SignalUnix.cpp:76-80
void SigintHandler(int signum, ...) {
    Cangjie::TempFileManager::Instance().DeleteTempFiles();   // 非异步信号安全
    _exit(signum + 128);
}

DeleteTempFiles()（TempFileManager.cpp:431）经 RemoveDirRecursively 调用 opendir/readdir（内部 malloc）及递归 —— 均非异步信号安全。对照崩溃路径正确使用的 DeleteTempFilesSignalSafe()（:464），此处明显遗漏。

影响

在编译期间按 Ctrl-C，若信号恰好在主线程持有 malloc 锁时送达，处理器内的 opendir/readdir 可能死锁；或产生堆状态不一致 / 清理不完整。非攻击者可控、用户自发触发，故为 Low。

修复

SigintHandler 改调 DeleteTempFilesSignalSafe()。

疑似发现 (Suspected Findings)

[VUL-003] syscap JSON 解析无递归深度上限（HIGH-06 残留项） Low-Med SUSPECTED

  • 位置：src/Sema/Plugin/ParseJson.cpp（ParseJsonObject ↔ ParseJsonArray 互递归），触发于 PluginCustomAnnoChecker.cpp:316/334
  • 已修复部分：数字解析改用安全 Stoull().value_or(0)；.back() 在 mod==VALUE 分支前由 : 处理器（:121-123）保证非空，转义处理重写 —— 故 HIGH-06 的 release UB 与数字异常已闭合。
  • 残留：解析器无递归深度计数器。--syscap=evil.json 提供 {"a":{"a":{...}}}（数万层，文件仅数十 KB）→ 栈溢出 → SIGSEGV（DoS）。
  • 附带：PluginCustomAnnoChecker.cpp:324-327 将首层 JSON 的 deviceSysCap 字符串值当作文件路径用 ReadBinaryFileToBuffer 读取 —— 配置内容可控编译器读取哪些文件（内容不外泄，仅作 JSON 再解析）。
  • 缺证据：syscap 配置在目标威胁模型（OpenHarmony 应用构建）中是否真由不可信方提供尚需确认。
  • 修复：ParseJsonObject/Array 增加 depth 参数与上限（如 256），超限返回诊断；对 deviceSysCap 路径做规范化/根目录约束。

[VUL-004] 裸工具名经 posix_spawnp 走 PATH 搜索 Low SUSPECTED

  • 位置：src/Driver/Tool.cpp:243-245（R3：需攻击者控制 PATH/env）
  • 当外部工具 name 不含目录分隔符时用 posix_spawnp（PATH 搜索）。环境构建 BuildEnvironmentVector（:259-318）已用黑名单剔除 LD_PRELOAD/LD_AUDIT/DYLD_*/RUSTC_WRAPPER 等加载器变量（MED-08 已缓解），但 PATH 本身被继承。在构建农场 / IDE / sudo 构建等 PATH 受影响场景，可能执行被植入的同名链接器/汇编器。
  • 残留点：env 过滤是黑名单而非白名单（历史推荐白名单），可能遗漏未来新增的危险变量。
  • 修复：链接器/汇编器一律解析为可信安装根下的绝对路径并用 posix_spawn；env 改白名单。

[VUL-005] GCC 版本串 std::stoi 无异常处理 Low SUSPECTED

  • 位置：src/Driver/Toolchains/GCCPathScanner.cpp:103/126/145

解析本地 GCC 版本号 std::stoi(versionStr.substr(...))，无 try/catch。畸形/超长版本串（本地 toolchain 被篡改）可抛异常崩溃。属 MED-13 同类、信任度更高（本地工具链），故 Low。改用 StdUtils::Stoi。
