1. 期望发生什么

对 release/1.0 分支来说，期望结果不是“所有复杂泛型场景都已经完美覆盖”，而是：已进入版本质量看护的 HLT 用例不应出现编译器 ICE；修复型 PR 在新增 CHIR 内部对象时，应保证对象生命周期、查询链路和回归测试闭环。
期望项 具体标准 为什么重要
版本质量 release/1.0 上 HLT/LLT 中与泛型、extend、mut wrapper 相关的既有用例应稳定通过，不能因内部转换缺口触发 ICE。 ICE 会直接阻断用户编译，是编译器质量中需要优先收敛的问题。
修复完整性 像 AddIndirectExtend 这类会动态生成 CHIR 对象的改动，必须确认对象被后续 package、type 查询、vtable、wrapper、CodeGen 路径一致识别。 CHIR 内部对象只接入局部消费路径，容易形成隐蔽的不一致状态。
测试覆盖 修复 PR 除原始复现用例外，应补充机制型用例，验证内部对象注册和后续查询链路。 只覆盖用户表象，不能证明底层机制已经闭环。
流程闭环 修复合入后，issue 状态、测试 PR、修复 PR 和版本标签应保持一致。 避免“代码已修、问题单仍待办”的追踪偏差。

3. 造成差异的原因是什么

3.1 #1354 引入的问题：新增 ExtendDef 生命周期未闭环

PR #1354 新增 AddIndirectExtend，在 SetVTable 阶段动态生成新的 ExtendDef 并生成 vtable。问题在于：原有 SetExtendInfo 只注册包内已有 extend， 对 #1354 新生成的 ExtendDef 不生效。结果是 vtable 生成路径能看到该对象，但 extended type 的查询路径看不到该对象，内部模型状态不一致。

直接差异：期望新增 CHIR 对象完成注册和查询链路闭环；实际只接入了 vtable 消费路径，未接入 extended type 反向索引。

3.2 历史存量问题：局部泛型函数实例化翻译顺序

原逻辑先把泛型函数声明放入待翻译列表，再收集实例化函数节点。lambda 或嵌套函数中调用局部泛型实例化函数时， 可能导致泛型原始版本先被翻译，实例化函数未按预期优先进入 CHIR 翻译。

3.3 历史存量问题：mut wrapper 继承路径计算

旧逻辑依赖共享 visited、stop 和回溯式 parents 记录继承路径。 在多接口、多继承链和泛型替换组合下，容易收集到错误路径或错误替换表，影响 mut 方法 wrapper 的函数类型实例化。

3.4 流程层面的差异

  • #1354 有测试 PR 并通过门禁，但测试重点更偏原始运行时崩溃场景，没有覆盖“新增 ExtendDef 被后续 type-based lookup 命中”的机制链路。
  • #1354 合入前曾因评审评论数量不足被拦截，后续通过 bypass 继续推进。该动作可以接受，但缺少更具体的检视依据沉淀。
  • #917 修复 PR 已合入，但 issue 页面仍显示 open/待办，说明问题状态闭环还可以改进。

4. 我们从中学到了什么

技术层面

  • CHIR 内部对象的生命周期应作为 review 的独立对象，而不仅是看单个算法是否正确。
  • 新增 IR/CHIR def 时，必须同时检查创建、归属、索引、查询、vtable、wrapper、CodeGen 等路径。
  • 泛型继承、extend、mut wrapper、局部泛型函数是组合风险区，单点测试不足以覆盖交互路径。

流程层面

  • 一个 issue 中出现多个 ICE 时，应先按触发机制拆分，再分别归因和验证。
  • 门禁通过说明基础质量达标，不等于语义路径已经完整覆盖。
  • 修复闭环应包含代码、测试、issue 状态和版本标签，而不只是 PR 合入。
本次复盘的主要收益是沉淀出一个可复用检查点：凡是动态生成 CHIR 对象，都要确认对象生命周期闭环，而不是只验证当前消费路径能跑通。
